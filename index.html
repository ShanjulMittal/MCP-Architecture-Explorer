High-Level Architecture

MCP operates on a flipped client-server model tailored specifically for AI. It acts as the "USB-C of AI", standardizing how models connect to the outside world.

MCP Host

The overarching AI application (e.g., Claude Desktop, Cursor IDE). It manages the UI, user intents, and orchestrates the AI model's generation process.

MCP Client

Embedded within the Host. It manages 1:1 connections to servers, handles protocol negotiation, and translates LLM tool-calls into JSON-RPC requests.

MCP Server

A lightweight, intermediary process. It exposes a standardized manifest of capabilities (Tools, Resources, Prompts) to the connected client.

Data Source

The actual backend system being accessed (PostgreSQL database, local filesystem, Slack API, GitHub webhooks, etc.).

Transport Layer

MCP uses JSON-RPC 2.0 as its wire format, but abstracts the transport mechanism to support both local execution and remote cloud connections.

1. Standard Input/Output (stdio)

The primary transport for local integrations. The AI Host spawns the MCP Server as a subprocess. Communication happens directly over `stdin` and `stdout`.

  • Extremely fast and synchronous.
  • Inherently secure (no network ports opened).
  • Perfect for local file system access, local database querying, or local IDE integration.

2. Server-Sent Events (SSE) & HTTP POST

Used for remote or cloud-based MCP Servers. Because traditional HTTP is stateless, MCP splits the communication:

  • Server-to-Client: Pushes notifications, progress, and responses via a persistent SSE stream.
  • Client-to-Server: Sends JSON-RPC requests via standard HTTP POST endpoints.

Client Capabilities

In traditional APIs, the client only makes requests. In MCP, the Server can request powerful capabilities from the Client (the LLM Host).

1. Sampling (The Agentic Loop)

Allows an MCP server to request an LLM completion from the client. This enables servers to utilize AI for intermediate processing before returning data to the main user context.

Use Case: A "News Aggregator" server fetches 5 articles. Before returning them, it calls client.sample() to ask the LLM to summarize the 5 articles into a single bias-free paragraph. The server then returns only the summary as the Resource.

2. Roots

Defines the spatial boundaries of what the server is allowed to access. The server queries the client to ask, "Where am I allowed to operate?"

Use Case: A local filesystem server needs to know which folder the user has currently opened in their IDE (Host). It requests roots, and the client responds with file:///Users/dev/my_project. The server restricts read/write operations to this directory.

Advanced Protocol Mechanics

To build robust, production-ready AI integrations, the protocol includes several advanced mechanics to handle large data, multimedia, and debugging.

{/* Pagination */}

1. Pagination (Cursor Management)

When an MCP Server exposes thousands of tools or resources, it cannot return them all at once without overwhelming the client. The protocol supports native pagination using cursor and nextCursor.

The client will pass this base64 cursor back in the next request to fetch page 2.

{/* Logging */}

2. Standardized Server Logging

Because MCP Servers often run as invisible subprocesses (via stdio), standard print statements get lost. MCP provides a dedicated notification endpoint for servers to stream logs (Info, Debug, Error) back to the Host's debugging UI.

{/* Multi-modal Content */}

3. Multi-modal Content (Images)

Tools and Resources are not limited to text. The `content` array returned by a tool can mix text with base64-encoded binary image data, allowing AI to analyze generated charts, screenshots, or diagrams.

{/* Ping Keep-Alive */}

4. Connection Liveness (Ping)

Especially for SSE/HTTP connections, either the client or the server can issue a standard ping request. The recipient must immediately respond with an empty result object to prove the connection is still alive.

); const LifecycleSection = () => (

State & Lifecycle

MCP is stateful. Connections follow a strict handshake and lifecycle pattern to ensure capabilities are mutually understood.

1. Initialization

Client sends initialize containing its capabilities (Roots, Sampling) and client info. Server responds with its protocol version and capabilities (Tools, Resources, Prompts).

2. Notification

Client sends an notifications/initialized event. This confirms the handshake is complete and the connection is ready.

3. Execution (Standard Operation)

Bi-directional flow. Client sends tools/call, server sends notifications/progress, server requests sampling/createMessage.

4. Shutdown

Client sends shutdown request. Server begins graceful termination, cleans up resources, and closes the stdio/SSE stream.

Security & Trust

Bridging autonomous LLMs with enterprise databases and local filesystems carries immense risk. MCP handles this at the protocol level.

Process Isolation

MCP servers run as isolated processes. A GitHub MCP server has no inherent access to a local filesystem unless explicitly granted by the Host. If an LLM is hijacked via prompt injection, the blast radius is strictly limited to the specific tools exposed by the active MCP server.

Human-in-the-Loop

MCP formally distinguishes between Safe tools (read-only) and State-Changing tools (e.g., executing SQL, sending emails). Client Hosts are required by design patterns to intercept state-changing tool executions and present a confirmation dialog to the human user before sending the JSON-RPC request.
